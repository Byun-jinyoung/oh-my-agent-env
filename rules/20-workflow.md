# 작업 절차 (20-workflow)

## Spec Gate — 코드보다 명세 먼저

- 새 프로젝트: 단계별 인터뷰 먼저 → `PROJECT.md` 작성·확정 → 그 다음 코드.
- 새 기능 / 모호한 요청: 인터뷰 먼저, 명세 둘째, 코드 셋째.
- 목표·제약·성공 기준·검증 방법이 분명해지기 전에는 코드를 작성하지 않는다.
- 불명확한 의도에서 시작/설계/구축하라는 요청에는 `spec-interview` 스킬을 사용한다.

## 의사결정 지원

- 분석/구현 전에 왜 필요한지, 어떤 문제를 해결하는지, 결과가 어떻게 쓰일지 설명한다.
- 분석 방향과 선택 이유를 밝힌다.
- 사용자가 결정해야 할 사항마다 선택지와 기준을 제시한다.
- 각 선택지는 배경, 의도, 장단점, 선택 기준을 포함한다.
- 추천은 근거, 제약, 비용, 위험, 되돌릴 수 있는지, 검증 비용을 기준으로 한다.
- 정보가 부족하면 선택을 막는 질문만 제시한다.

## 코드 변경 전 분석

- 관련 파일과 entry point를 먼저 찾는다.
- 데이터 흐름을 입력, 처리, 출력으로 나눠 파악한다.
- 중요한 데이터 구조와 타입을 확인한다: JSON/YAML/TOML 키, CLI 인자, 파일 구조, record 수, 크기 등.
- 기존 동작에 영향을 주면 현재 동작을 먼저 관찰한다.
- 구현 경로가 여러 개면 근거, tradeoff, 검증 비용으로 선택 이유를 설명한다.
- 문제 진단은 한 덩어리로 다루지 말고 **단순화·분해하여 여러 측면**에서 검사한다. 큰 문제는 가능한 한 작은 하위 문제로 쪼개 각각을 독립적으로 진단·재현하고, 그 결과를 합쳐 원인을 좁혀 간다.
- 분해한 하위 문제는 **단계화된 ToDo list**로 추적한다. 한 번에 하나씩 진행하고 완료 즉시 표시해 무의미한 반복·중복 작업을 방지한다.
